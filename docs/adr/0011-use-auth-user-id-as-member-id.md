# Use Auth User ID as Member ID

CSK Choir Hub will use the linked Auth User's identifier as the `Member.id` value. The `Member` model will not store a separate `userId` column: a Member is linked to its Auth User by shared identity, and all Group Membership and Position Assignment records reference that shared Member ID.

This decision supersedes the `userId` scalar link described in ADR-0003. It keeps the existing ownership boundary—choir-domain Member data remains separate from the autogenerated auth User model—while making the already-required one-to-one identity relationship explicit in the identifier itself.

Member creation must receive an existing or newly created Auth User ID and persist it as `Member.id`. Current-actor domain predicates can therefore use the authenticated session's User ID directly as the Member ID without an intermediate lookup.

The v1 database may be recreated during development, so no compatibility migration is required for the removed `userId` column. Future account replacement or account-merging workflows must deliberately preserve or reassign the shared identity and its dependent domain records rather than assuming Member IDs are independent of Auth User IDs.
